## 定义

规范架构（Normative Architecture）是一种架构设计方法，其核心理念是通过强制执行不变量（invariants）而非对实施过程进行微观管理，使智能体能够快速交付且不削弱基础。

## 核心原则

### 强制执行不变量，而非微观管理

- 要求智能体在边界处解析数据形状，但不规定具体实现方式
- 例如：要求使用"解析而非验证"模式，但不指定具体库（如Zod）
- 保持智能体在解决方案表达方式上的自由度

### 严格的架构模型

- 每个业务域划分为一组固定的层
- 依赖方向经过严格验证
- 仅允许有限的一组边
- 通过自定义linter和结构测试机械地强制执行

### 分层示例

在每个业务领域内（例如应用设置），代码只能"向前"依赖于一组固定的层：
- Types → Config → Repo → Service → Runtime → UI

横切关注点（认证、连接器、遥测、功能标志）通过一个单一的显式接口进入：Providers。

## 实践要点

### 早期先决条件

- 这种架构通常要等到拥有数百名工程师时才会推迟
- 对于编码智能体来说，这是一个早期的先决条件
- 有了约束，速度才不会下降，架构才不会漂移

### 自定义工具

- 通过自定义的代码检查器和结构测试来强制执行规则
- 编写错误信息时会在智能体情境中注入修复指令
- 规则一旦编码，就能立即应用于所有地方

### 品味不变式

- 结构化日志记录
- 模式和类型的命名约定
- 文件大小限制
- 特定平台的可靠性要求

## 人类的角色

- 在中央层面强制执行边界
- 在本地层面允许自主权
- 重视界限、正确性和可重复性
- 允许智能体在边界内拥有解决方案表达的自由
